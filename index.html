Why anyother named array package?

  • Geospatial raster data are essentially 2-N dimensional arrays
  • They have named dimensions and lookup values
  • There are a lot of lookup configurations
  • Multi-array datasets that share some or all dimensions are common
  • But other data follows similar patterns
  • So make it generic and extensible to maximise contributions

Dimensions

Wrappers types

Dimension wrappers mark an object belongs to a dimension:

using DimensionalData 
X(1)
X 1


X(50:10:100)
X 50:10:100


X(Not(At(70)))
X InvertedIndices.InvertedIndex{At{Int64, Nothing, Nothing}}(At(70, nothing, nothing))

They can be used in dims keywords:

mapslices(sum, A; dims=X)

But Symbols work too there:

mapslices(sum, A; dims=:X)

“Standard” dimensions (90% of spatial data):

X, Y, Z, Ti
(X, Y, Z, Ti)

Arbitrary dimensions (everything else)

Dim{:name}
Dim{:name}

Lookups

  • Lookup <: AbstractArray arrays that hold another AbstractArray with lookup metadata.
    • points or intervals
    • sampled or categorical.
    • Lookups are extensible, Rasters.jl lookups have a coordinate reference system
  • These properties change how selectors work for both correctness and performance
  • Mostly, lookups are generated automatically by any AbstractDimArray constructor,
    • you can ignore them until you need something specific.

Named indexing

No-cost abstractions

Selectors

Fast lookups


A = DimArray([1, 2, 3, 4], (X([10.0, 20.0, 40.0, 80.0])))
A[X(At(80.0))]
A[X(Near(85))]
4

Lookups over Intervals


using DimensionalData.Lookups
A = DimArray(100:100:9900, X(1.0:1.0:99.0; sampling=Intervals(Start())))
A[X(Contains(9.5))]
900
